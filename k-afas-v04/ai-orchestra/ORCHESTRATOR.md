# AI 오케스트라 시스템 - 총괄 지휘 매뉴얼

## 핵심 원칙
```
토큰 절약 = 역할 분리 + 컨텍스트 압축 + 적재적소 AI 배치 + 재귀검증 2회 이내
```

## 2. 역할별 AI 배치 (확정)

| 역할 | 담당 AI | 조건 | 비용 |
|------|---------|------|------|
| **🏗️ 설계 (주력)** | Claude Sonnet 최신 | 모든 설계 작업의 기본 | 구독 내 |
| **🏗️ 설계 (극제한)** | Claude Opus 4.7 | 반드시×2 필요할 때만, 핵심 부분만 | 구독 내 |
| **🔍 설계 검증** | ChatGPT 최신 (GPT-5.5) | 설계 결과를 검증/반박 | 구독 내 |
| **🔧 구현** | GPT-5 mini / Gemini Flash | 코드 작성, 수정, 반복 | $0.25~$0.50/MTok |
| **🔧 구현 (복잡)** | Kiro (Sonnet) | 여기서 직접 | 포함 |

### ⚠️ 핵심 규칙
```
설계: Claude Sonnet 주력. Opus는 "반드시 반드시" 필요할 때 핵심 부분만.
검증: ChatGPT 최신. 설계를 검증/반박. 재귀 최대 2회.
구현: mini/Flash 저비용 AI. 또는 Kiro에서 직접.
```

## 3. 재귀적 검증 프로토콜 (최대 2회)

```
┌─────────────────────────────────────────────┐
│         재귀적 검증 흐름 (MAX 2회)            │
├─────────────────────────────────────────────┤
│                                             │
│  [설계] Claude Sonnet → 설계서 출력          │
│     │                                       │
│     ▼                                       │
│  [검증 1회차] ChatGPT → Pass/Fail + 이유    │
│     │                                       │
│     ├─ Pass → 구현 단계로 진행               │
│     │                                       │
│     ├─ Fail → 수정사항 명시                  │
│     │     │                                 │
│     │     ▼                                 │
│     │  [수정] Claude Sonnet → 수정 설계      │
│     │     │                                 │
│     │     ▼                                 │
│     │  [검증 2회차] ChatGPT → Pass/Fail     │
│     │     │                                 │
│     │     ├─ Pass → 구현 진행                │
│     │     └─ Fail → 사용자 판단 요청 (강제종료)│
│     │                                       │
│     └─ 3회 이상 절대 금지 (토큰 낭비)         │
└─────────────────────────────────────────────┘
```

### 검증 프롬프트 (ChatGPT에 복사용):
```
아래 설계를 검증하라. Pass/Fail만 판단.

[설계서]
{설계 내용 - L1 압축}

출력 형식 (이것만):
- 판정: Pass 또는 Fail
- Fail 시 이유: 1~3줄 (구체적 문제점만)
- 수정 제안: 1~2줄

제약: 100토큰 이내. 칭찬/설명 금지.
```

## 4. Opus 4.7 사용 하네스 (극도 제한)

> **기본: 쓰지 않는다.** Sonnet으로 99% 해결.

```
┌─────────────────────────────────────────────────────────────┐
│  💎 Opus 사용 허가 게이트 (4개 ALL YES + 핵심 부분만)         │
├─────────────────────────────────────────────────────────────┤
│ ① Sonnet으로 2회 시도했으나 불충분한가?      → YES 필수     │
│ ② 시스템 레벨 복잡도인가? (100+파일)         → YES 필수     │
│ ③ 실패 시 전체 장애 위험인가?                → YES 필수     │
│ ④ 설계 전체가 아닌 "핵심 판단 부분만"인가?   → YES 필수     │
└─────────────────────────────────────────────────────────────┘
→ 하나라도 NO = Sonnet 사용
→ 통과해도: 설계 전체 X, 핵심 아키텍처 결정 부분만
```

## 5. 전체 작업 흐름 (Step-by-Step)

```
[요청 접수]
    │
    ▼
[Phase 1: 설계] → Claude Sonnet (별도)
    │              Kiro가 초압축 프롬프트 제시 → 사용자 복사
    │              (Opus: 극제한 게이트 통과 시에만)
    ▼
[Phase 2: 검증] → ChatGPT 최신 (별도)
    │              Kiro가 검증 프롬프트 제시 → 사용자 복사
    │              재귀 최대 2회 → Fail 시 사용자 판단
    ▼
[Phase 3: 구현] → Kiro(직접) 또는 mini/Flash 프롬프트 제시
    │              코드 작성, 수정
    ▼
[완료]
```

## 6. 토큰 극소화 전략

### 모든 프롬프트에 적용:
```
- 컨텍스트: L2 압축 (100토큰 이내)
- 지시: 1~2문장
- 출력 형식: 강제 (하네스)
- 토큰 상한: 항상 명시
- 설명/서론/결론: 금지
- 입력 프롬프트 목표: 300토큰 이내
- 출력 상한: 500토큰 이내 (설계), 100토큰 이내 (검증)
```

### 컨텍스트 압축 필수 적용:
```
L0 (원본)  → 현재 수정 중인 코드만
L1 (요약)  → 시그니처 + docstring
L2 (메타)  → "파일명: 역할 1줄"
L3 (태그)  → "폴더: N개 파일 존재"
```

## 7. 세션 관리 규칙

| 조건 | 행동 |
|------|------|
| 주제 30개 도달 | → SESSION_HANDOFF.md 실행 |
| 컨텍스트 70% 소진 | → CONTEXT_COMPRESSOR.md 실행 |
| 설계 필요 | → Claude Sonnet 프롬프트 생성 |
| 설계 검증 필요 | → ChatGPT 검증 프롬프트 생성 |
| 구현 필요 | → Kiro 직접 수행 또는 mini/Flash 프롬프트 |
| 재귀검증 2회 Fail | → 사용자에게 판단 요청, 강제 진행 |

## 8. Kiro 행동 규칙 (이 세션부터 적용)

```
1. 설계 필요 시:
   → "Claude Sonnet에 복사하세요:" + 초압축 프롬프트 제시
   → (극제한) Opus 필요 판단 시 사전 고지 + 게이트 체크

2. 검증 필요 시:
   → "ChatGPT에 복사하세요:" + 검증 프롬프트 제시
   → 재귀 최대 2회, 2회 Fail 시 사용자 판단 요청

3. 구현 필요 시:
   → Kiro에서 직접 수행 (대부분)
   → 또는 mini/Flash 프롬프트 제시

4. 모든 프롬프트:
   → 컨텍스트 L2 압축
   → 입력 300토큰 이내
   → 출력 상한 명시
   → 하네스 형식 강제
```
